# Composition Rules

Правила композиции **экранов** Larixon Mobile DS. Не про компоненты — про то, как компоненты складываются вместе.

> Спеки компонентов (`*-spec.md`) описывают внутреннее устройство компонента: его токены, варианты, состояния. Правила композиции описывают, **как несколько компонентов собираются в экран** — отступы между ними, иерархия, контейнеры, размерная консистенция.

**Источник правды:** этот файл (`main` ветка mdsl). Любые другие копии (Figma-плейбук, SKILL.md ассистентов, Confluence-страницы) — производные. При расхождении верить этому файлу.

**Связанные документы:**
- [DESIGN-TOKENS.md](./DESIGN-TOKENS.md) — primitives (`spacing/N`, `radius/N`, `size/N`)
- [spacing-semantic.md](./spacing-semantic.md) — семантические токены отступов (`section/*`, `stack/*`, `row/*`, `chip/*`, `cta/*`, `screen/*`, `list/*`)
- [TYPOGRAPHY.md](./TYPOGRAPHY.md) — текстовые стили и роли
- [COLOR-PALETTE.md](./COLOR-PALETTE.md) — семантика цвета

## 1. Header = StatusBar + Top App Bar — неразделимы

**Правило:** StatusBar и Top App Bar — один логический блок «Header». Они всегда идут парой, без `gap` между ними. Никакая фоновая полоса, разделитель или вертикальный отступ между ними не вставляется.

**В Figma:** обёрнуты в общий `VERTICAL` frame **«Header»** с `itemSpacing = 0` и подаются как **один** child экранного контейнера.

```
Screen (VERTICAL)
├── Header (VERTICAL, gap = 0)           ← один child
│   ├── StatusBar
│   └── Top App Bar
├── Section 1
├── Section 2
└── ...
```

**Анти-паттерн:**
```
Screen → [StatusBar, Top App Bar, Section1, ...] с screen.itemSpacing = 8
                                ↑ разрыв между StatusBar и Top App Bar
```

Подробнее — в [top-app-bar-spec.md](./top-app-bar-spec.md) § «Композиционное правило».

## 2. Section card — поверхность с радиусом

**Правило:** каждая контентная секция экрана — поверхность (Card) с фоном `Surface/Surface Primary` и `radius/surface/surface` (12). Углы становятся видны на стыке с `Background/Secondary` через `section/gap`.

**Не делать:** секции без радиуса, плотно лежащие друг к другу через 1px-divider. Это веб-паттерн, не мобильный.

**Density tiers секции** (выбираются по типу контента):

| Tier | `padding` | Когда |
|---|---|---|
| Compact | `section/padding-compact` (16) | Списки фильтров, цепочка однотипных блоков, выбор языка |
| Default | `section/padding-default` (24) | Профиль, описание, payment, обычный контентный блок |
| Spacious | `section/padding-spacious` (32) | Hero, финальный экран с одним действием |

Тир секции диктует размер компонентов внутри (см. § Размерная консистенция).

## 3. Section header — proximity rule

**Правило (proximity):** section heading композиционно «приклеивается» к своему content'у — пространство между ним и его content'ом **меньше**, чем пространство между ним и предыдущей секцией. Это работает в обоих layout-моделях:

- **3.1 Card layout** — heading как первая row внутри card
- **3.2 Flat layout** — heading с asymmetric padding между секциями

Выбор между card- и flat-моделью — **по source**, а не по предпочтению (см. §3.3).

### 3.1 Card layout — heading первой row внутри карты

Когда секция обёрнута в card, заголовок становится **первой row внутри card**, не отдельный элемент над ней.

**Структура:**

```
Card (section/padding-compact = 16)
├── Heading «Все марки»                ← section header
├── ↕ section/title-to-content (8)
└── Content
    ├── Row 1
    ├── Row 2
    └── ...
```

**Что это даёт:**
- Сверху от Heading: 16 (внутренний padding карты)
- От Heading до Content: 8 (`section/title-to-content`)
- Снизу от последней Row: 16 (внутренний padding карты)

Симметрично с другими card'ами на экране — все имеют одинаковый внутренний воздух 16, не зависит от того, есть в карте заголовок или нет.

**Анти-паттерн:** Heading как отдельный блок над Card.

```
[Heading «Все марки»]   ← 0 внутреннего padding, странно висит
[Card с Content]        ← 16 padding, диссонанс
```

### 3.2 Flat layout — heading с asymmetric padding

Когда секции не обёрнуты в card'ы (длинные плоские списки, плотные мобильные экраны), заголовок секции стоит **между rows прошлой и своей секции** с **asymmetric padding**: внешнее (heading → предыдущая секция) больше, чем внутреннее (heading → свой content).

**Структура:**

```
List item (последний из предыдущей секции)
                ↕ 24 (внешнее, spacing/6)     ← отделяет от предыдущей секции
«Section heading» (text)
                ↕ 8 (section/title-to-content) ← биндит к своему content
List item (первый из своей секции)
```

**Значения:**
- **Внутреннее (M):** 8 — `section/title-to-content` (существующий токен)
- **Внешнее (N):** 24 — `spacing/6` напрямую. Если паттерн закрепится — кандидат на отдельный токен `section/heading-to-prev`.

**Что это даёт:** heading визуально принадлежит своей секции, а не «висит ничей» между секциями. Закон Gestalt-близости (proximity): элементы ближе друг к другу воспринимаются как одна группа.

**Анти-паттерн (симметричный sandwich):**

```
List item
       ↕ 8
«Heading»
       ↕ 8
List item
```

Heading сэндвич-зажат с равными зазорами — proximity-сигнал пропал, визуально не принадлежит ни одной секции. Это **тот самый bug**, который ds-build допустил в PB-876 при пересборке мультиселекта моделей машин («Популярные модели» / «Все модели») — секции оказались без визуальной группировки.

### 3.3 Card vs flat — выбор по source, не по preferences

**Правило:** выбор между card-обёрткой секции и flat-структурой определяется **source-материалом**, не отдельным дизайнерским решением исполнителя.

- Если source оборачивает секции в card'и → собирай card'ы (применяй §3.1)
- Если source использует flat-структуру → не вводи card'ы (применяй §3.2)
- **Не изобретать** card'и поверх flat source'а
- **Не снимать** card'и с card source'а

**Когда правило поднимается:** в начале пересборки экрана, при анализе source-композиции. Это первый вопрос rebuild-флоу, до раскладки rows.

**Что это даёт:** consistency между source intent и result. Если source-дизайнер выбрал flat — у него были на то основания (длинный список, плотность, мобильный контекст). Rebuild не должен переинтерпретировать структуру.

**Контекст возникновения правила:** ds-build при пересборке PB-876 переинтерпретировал flat-source как card-layout и применил §3.1 буквально (heading первой row внутри card). Result не соответствовал source intent. Правило §3.3 — guardrail против этой регрессии.

## 4. Разделение секций — gap, а не sep-frame

**Правило:** между соседними section card используется `screen.itemSpacing = section/gap` (8). Фон `Background/Secondary` проступает в 8 пикселей и работает разделителем.

**Sep-frame** (явный визуальный объект между секциями — 16-px полоса другого цвета, баннер, индикатор) — допустим, **если у него есть собственная семантика**, а не просто «mock-gap».

**Анти-паттерн:** sep-frame с фоном = фон экрана + фиксированной высотой 8. Это маскирует то, что должно быть просто gap.

## 5. Размерная консистенция компонентов в секции

**Правило:** компоненты в одной секции — одного размерного тира.

| Tier | Chips | Input v2 | Button | Когда |
|---|---|---|---|---|
| Compact | 32 | sm | 32 / 40 | Плотная подача, фильтры с большим числом опций |
| Default | 40 | md | 48 | Filter/form-секции (по умолчанию) |
| Spacious | — | lg | 56 | Hero CTA, финальный экран |

**Анти-паттерн:** Chips 40 + Input lg → инпут выше чипсов на ~8px, ритм ломается.

**Решающий тест:** если сомневаешься — выбирай Default (md / 40).

## 6. Title screen ↔ Subtitle — stack/gap-tight (4)

**Правило:** заголовок экрана и подзаголовок-описатель читаются как одна логическая единица. Между ними `stack/gap-tight` (4), не `stack/gap-default` (8).

```
Title «Марки»
↕ 4 (stack/gap-tight)
Subtitle «Выбрано: 3»
```

Это распространяется на любой `Title + Subtitle` блок — в Hero, в Top App Bar (subtitle), в карточке.

## 7. Divider в списке — опционален

**Правило:** `list/item-gap = 0` (rows внутри списка вплотную) — да. Но **Divider между rows не обязателен**. По умолчанию — без дивайдера, пустота 0 между rows достаточна для визуального разделения.

**Дивайдер добавлять только когда:**
- Список очень плотный без явных границ строк и нужна сетка
- Разные категории внутри одного списка (group separator)
- Дизайн исходника явно использует дивайдер как часть визуального ритма

**Решающий тест:** посмотри на исходник. Есть линии между rows? — ставим Divider. Нет линий? — не ставим.

**Divider — не часть List Item**, это композиционный элемент. Не существует variant'а List Item «с дивайдером».

## 8. Brand colors / нестандартные цвета компонентов — НЕ override

**Правило:** если в исходнике (скриншот, чужой Figma, мокап) CTA / badge / чип нашего компонента показан другим цветом, чем стандартный (например, **жёлтая Apply** вместо тёмной Primary, **brand-tinted checkbox** вместо стандартной) — **НЕ override**ить fill компонента.

**Использовать DS-вариант компонента как есть.** Расхождение записывать как DS-gap.

**Override fill допустим** только если пользователь явно требует «именно этот цвет» — тогда метим, что это временный визуальный «костыль».

**Исходник** — это **что** надо показать. **Наши правила** — это **как** мы это показываем.

## 9. Bottom CTA — через preset

**Правило:** кнопка фиксированного действия внизу экрана подаётся через один из трёх preset'ов.

| Preset | Padding top | Padding bottom | Когда |
|---|---|---|---|
| Default | `cta/padding-top-default` (16) | `cta/padding-bottom-default` (12) | Обычный экран с CTA |
| Spacious | `cta/padding-top-spacious` (24) | `cta/padding-bottom-spacious` (32) | Hero, финальный экран |
| Compact | 0 | 0 | Поверх клавиатуры (контекстный действующий слой) |

Горизонтальный padding — всегда `screen/padding-horizontal` (16).

---

## 10. Screen edge — screen/padding-horizontal = 16

**Правило:** боковые отступы любого контента от края экрана — `screen/padding-horizontal` (16). Применяется к контейнеру экрана, не к каждой секции отдельно.

**Исключение:** List Item — edge-to-edge (FILL контейнер 360 без horizontal padding), потому что master компонента имеет фиксированную ширину 360. Если нужны отступы у списка — оборачивай в Card с padding.

## Контракт композиции экрана

Каждый собираемый экран должен удовлетворять:

- [ ] Header (StatusBar + Top App Bar) — обёрнут в общий frame с gap=0
- [ ] Card vs flat — выбран по source, не изобретён (§3.3)
- [ ] Каждая section — card с `radius/surface/surface` и `section/padding-*` (если source — card)
- [ ] Заголовки секций в **card-layout** — первой row внутри card (§3.1)
- [ ] Заголовки секций в **flat-layout** — asymmetric padding, внешнее (24) > внутреннее (8) (§3.2)
- [ ] Между card'ами — `section/gap` (8), без sep-frame
- [ ] Компоненты внутри одной секции — одного размерного тира
- [ ] Title + Subtitle — `stack/gap-tight` (4)
- [ ] Divider в списке поставлен только если есть на исходнике
- [ ] Цвета компонентов — DS-варианты, без override fill
- [ ] Bottom CTA — через preset (Default / Spacious / Compact)
- [ ] Screen edge — `screen/padding-horizontal` (16)
