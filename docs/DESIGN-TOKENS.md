# Правила использования переменных (Design Tokens)

## Структура токенов

Переменные разделены на два уровня:

- **Core** — примитивные значения (числовая шкала). Не используются напрямую в экранах и компонентах.
- **Semantic** — семантические токены, ссылающиеся на Core. Используются во всех макетах и в коде.

В интерфейсе (код, превью) применяются только токены из **Semantic**.

---

## Core

### Spacing

- Базовая единица: **4 px**.
- Токен `spacing/N` = **4 × N** px.
- Индексы шкалы: 0, 1, 2, 3, 4, 5, 6, 8, 10, 12, 14, 16.

| Token       | Значение (px) |
|-------------|---------------|
| spacing/0   | 0             |
| spacing/1   | 4             |
| spacing/2   | 8             |
| spacing/3   | 12            |
| spacing/4   | 16            |
| spacing/5   | 20            |
| spacing/6   | 24            |
| spacing/8   | 32            |
| spacing/10  | 40            |
| spacing/12  | 48            |
| spacing/14  | 56            |
| spacing/16  | 64            |

### Radius

- Токен `radius/N` = **4 × N** px для N = 0…5.
- `radius/full` = 999 px (полное скругление, pill/круг).

| Token       | Значение (px) |
|-------------|---------------|
| radius/0    | 0             |
| **radius/0_5** | **2**       |
| radius/1    | 4             |
| radius/2    | 8             |
| radius/3    | 12            |
| radius/4    | 16            |
| radius/5    | 20            |
| radius/full | 999           |

**`radius/0_5` (2 px)** — введён 2026-05-11 для Material 3 Checkbox (M3-спека требует 2 px). Между `radius/0` и `radius/1`. Используется в маленьких прямоугольных контролах с минимальным скруглением (Checkbox box). Имя через `_` потому что точка `.` в имени переменной Figma не разрешена.

### Border (толщина линии)

- Шкала в **пикселях**, шаг 1 px (для обводок и разделителей).
- Токены толщины бордера и дивайдеров.

| Token      | Значение (px) |
|------------|---------------|
| border/1   | 1             |
| border/2   | 2             |

## Semantic

### space

Внутренние отступы и зазоры внутри компонентов (padding, gap между иконкой и текстом, между чипами и т.п.).

- Использовать для: padding кнопок, полей, карточек; gap внутри компонента.
- Не путать с **stack** — stack для вертикальных отступов между блоками.

### stack

Вертикальные отступы между секциями и блоками контента (между карточками, между заголовком и списком и т.д.).

- **stack-sm** → spacing/4 (16 px)
- **stack-md** → spacing/6 (24 px)
- **stack-lg** → spacing/8 (32 px)
- **stack-xl** → spacing/12 (48 px)
- **stack-2xl** → spacing/16 (64 px)

**Почему шкала начинается с `sm = 16`:** меньшие значения (8 / 12) — это уже не межблочное расстояние, а **внутренний padding контейнера** (см. `space`). Stack используется только когда блоки визуально различимы как отдельные сущности.

### control-height

Минимальная высота **интерактивных контролов** (кнопки, инпуты, селекты). Гарантирует touch target ≥32 px.

| Токен | Референс | Значение | Использование |
|-------|----------|----------|---------------|
| control-height/xs | spacing/8 | 32 px | Компактные кнопки (xs) |
| control-height/sm | spacing/10 | 40 px | Маленькие кнопки, компактные инпуты |
| control-height/md | spacing/12 | 48 px | Стандартные кнопки и инпуты |
| control-height/lg | spacing/14 | 56 px | Крупные CTA, большие инпуты |

- В компонентах: фиксированная высота фрейма привязана к переменной.
- Для текстарии используется отдельный минимум, не control-height/*.
- **Не использовать** для не-контролов (бейджи, аватары) — см. `size/*`.

**Почему шкала заканчивается на `lg = 56`:** значения 64+ px на мобильных контролах — это лендинговые/маркетинговые CTA, не системный паттерн. На основных экранах продукта таких размеров не должно быть.

### size

Универсальная шкала размеров для **не-контролов**: бейджи, аватары, иконки, thumbnails. Touch target не требуется, поэтому минимум — 16 px.

| Токен | Референс | Значение | Использование |
|-------|----------|----------|---------------|
| size/2xs | spacing/4 | 16 px | Микро-бейдж, dot-индикатор |
| size/xs | spacing/5 | 20 px | Каунтер, маленький бейдж |
| size/sm | spacing/6 | 24 px | Компактный бейдж, мини-аватар |
| size/md | spacing/8 | 32 px | Стандартный бейдж, аватар sm |
| size/lg | spacing/10 | 40 px | Аватар md, большой бейдж |
| size/xl | spacing/12 | 48 px | Аватар lg |
| size/2xl | spacing/14 | 56 px | Большой аватар |
| size/3xl | — (off-scale) | 80 px | Аватар 4XL (identity-illustration tier — см. ниже) |

> **`size/3xl` — off-scale, identity-tier constant** (зеркалит `illustration/md`). На этом узле шкала `size/*` покидает spacing-core: 80 px не используется как padding/gap в продукте, и плодить `spacing/18` / `spacing/20` ради единственного аватара — orphan-токены, провоцирующие «64 или 80?» при следующем макете. То, что `size/2xs..2xl` ложатся на `spacing/*` — следствие плотности нижней части шкалы, а не правило системы; на `3xl` size-шкала переходит из «layout dimension» в «identity asset».
>
> **Identity-illustration tier (64 / 80 px).** Avatar `3XL` (64 px) и `4XL` (80 px) — на границе между «информационным элементом» и «иллюстрацией». В Figma они подключены через `spacing/16` (64) и `size/3xl` (80, off-scale). Для разработки: на этих размерах аватар читается как identity-block, не как контрол; масштабирование иконок и шрифта идёт по illustration-логике, а не по control-логике. См. `docs/avatar-spec.md` §3.

**Когда использовать `size/*` vs `control-height/*`:**

| Элемент | Токен | Причина |
|---------|-------|---------|
| Кнопка, инпут, селект | `control-height/*` | Нужен touch target ≥32 px |
| Бейдж, аватар, иконка | `size/*` | Touch target не нужен |
| Интерактивный бейдж | Обёртка: `control-height/*`, бейдж внутри: `size/*` | Обёртка обеспечивает touch target |

### radius

Семантика скруглений (алиасы на Core, коллекция Semantic):

| Semantic (short) | Figma path                       | Core      | px  | Использование |
|------------------|----------------------------------|-----------|-----|----------------|
| control-sm       | `radius/control/control-sm`      | radius/1  | 4   | Мелкие контролы |
| control-md       | `radius/control/control-md`      | radius/2  | 8   | Кнопка **xs**, rounded badge (sm/md/lg/xl), мелкие кнопки-иконки |
| control-sm (badge 2xs/xs) | `radius/control/control-sm` (на badge) | radius/1 | 4 | Rounded badge **на 2xs/xs** — 4px (8px смотрелся бы почти как Pill на низкой плашке). См. badge-spec.md §2. |
| control-lg       | `radius/control/control-lg`      | radius/3  | 12  | Кнопки **sm**, **md**, **lg** |
| surface          | `radius/surface/surface`         | radius/3  | 12  | Карточки, панели |
| overlay          | `radius/surface/overlay`         | radius/4  | 16  | Модалки, bottom sheet (открытый край) |
| pill             | `radius/pill/pill`               | radius/full | 999 | Капсулы, чипы-pill |

> **Шкала `radius/surface/*`** — два токена (`surface` = 12, `overlay` = 16) под зонтиком одной семьи: разные уровни приподнятости поверхности дают разный «мягкий» радиус. Surface = карточка/панель (12), overlay = модалка/sheet (16) — у overlay край больше, потому что поверхность визуально «оторвана» от фона.

### border и divider (толщина)

| Токен              | Референс  | Использование |
|--------------------|-----------|----------------|
| border/default     | border/1  | Обводки элементов: карточки, инпуты, чипы |
| border/emphasis    | border/2  | Фокус, выделение, акцент; **Outline-кнопка** sm/md/lg (см. docs/button-spec.md) |
| divider/default    | border/1  | Разделители между блоками и строками списка |

**Border vs Divider — разные семантические роли**, даже при совпадающих значениях:

- **Border** — обводка элемента (контейнер имеет «оболочку»).
- **Divider** — линия в потоке между элементами (между строками списка, секциями).

Сейчас оба = 1 px, но в реализации **держим как отдельные токены** (`BorderWidth.default` и `Divider.default` в коде). Это запасной задел: если в будущем Divider уйдёт в hairline (0.5 px по платформенным гайдлайнам), правка делается в одной точке без обхода компонентов.

- Цвет линии задаётся отдельными цветовыми токенами (color/border, color/divider).
- **Кнопка Outline:** высота **32 (xs)** → толщина **border/default** (1 px); **40 / 48 / 56** → **border/emphasis** (2 px). Реализация: `borderWidth` в `@/tokens` (spacing.ts).

### typography (подписи, caption)

Токены подписей и мелкого текста (подробнее — **docs/TYPOGRAPHY.md**):

| Токен | Размер (px) | Line height (px) | Начертание |
|-------|-------------|------------------|------------|
| typography/caption-lg | 14 | 16 | Regular, Medium, Bold |
| typography/caption-md | 12 | 16 | Regular, Medium, Bold |
| typography/caption-sm | 10 | 12 | Medium |

Использовать в таблицах токенов компонентов для строки «Текст» (например typography/caption-lg, color/text-primary).

### Иконки

- **Размеры в UI:** **16**, **24** и **32** px — основная шкала. Других размеров не вводить как general-purpose.
- **Исключение — Badge 2xs/xs:** микро-иконка **12 × 12** px. Реализуется **масштабированием 16-asset до 12** (SVG lossless, без отдельной 12-шкалы в библиотеке). Применяется только в badge-spec.md §3 на размерах 2xs (16 px высота) и xs (20 px высота), где 16-px иконка не помещается с padding'ом. На остальных компонентах 12-px иконку не использовать.
- В спецификации компонента для иконок указывать: размер (12 — только Badge 2xs/xs / 16 / 24 / 32) и цвет (семантический токен).
- Источник — Larixon Mobile библиотека, имена с префиксом `ic_*` (legacy без префикса не использовать).
- **Если нужен «крупный icon-asset» (44 / 64 / 80 px и больше)** — это **Illustration**, не Icon. См. ниже.

### Illustrations

Графика крупнее 32 px — это иллюстрации, не иконки. Источник — отдельная папка illustrations, не библиотека `ic_*`.

| Token | px | Использование |
|---|---|---|
| illustration/sm | 64 | Action chips, info-блоки, **крупный профильный аватар (XXXL)** |
| illustration/md | 80 | Empty state, **крупный профильный аватар (XXXXL)** |
| illustration/lg | 120 | Большие декоративные illustrations |

- В спецификации компонента, если используется illustration, указывать токен размера (`illustration/sm` и т.д.) и source-asset.
- **Промежуточные размеры (44–56 px)** в системе **не используются**. Если возникает потребность — это либо `size/2xl` для декоративного элемента (56 px), либо иконка 32 px внутри touch-target большего размера (см. `control-height` / `size`).
- **Крупные профильные аватары** (XXXL=64, XXXXL=80) семантически принадлежат к этой шкале как identity-illustration (profile-header, settings, чат-инфо). Не интерактивны, играют роль visual anchor, не control. Не путать со шкалой `size/*` для control-height.

## Правила использования

1. **На уровне экранов и страниц** (layout, между блоками контента) — использовать **только Semantic**: `stack/*` для вертикальных отступов между блоками, `radius/*`, `border/*` и т.п. Core (`spacing/N`) на уровне экранов **не подставлять**.

   **Внутри компонента** (padding контейнера, gap между внутренними элементами) — допустимо ссылаться на **`spacing/N` напрямую** из Core. Это семантически и есть «space», просто без промежуточного алиасинга — внутренние отступы компонентов слишком разнообразны для пресет-шкалы (например, кнопка использует spacing/4 horizontal, spacing/3 vertical, spacing/2 gap).
2. **Отступы**: внутри компонентов — **space** (= `spacing/N` напрямую); между блоками по вертикали — **stack/\***.
3. **Высота контролов**: у кнопок и однострочных инпутов — **control-height/xs** (32), **control-height/sm** (40), **control-height/md** (48) или **control-height/lg** (56). Один размер в одном контексте (например, основная кнопка и поле ввода в одной форме — control-height/md).
4. **Размеры не-контролов**: бейджи, аватары, иконки — **size/2xs** (16) … **size/2xl** (56). Не использовать control-height для информационных элементов.
4. **Скругления**: выбирать из **radius** по смыслу (control-*, surface, overlay, pill), не вводить произвольные значения.
5. **Текстария (многострочное поле)**: минимальная высота задаётся отдельным токеном (например control-height/textarea-min), не control-height/sm|md|lg. Фиксированная высота у текстарии не используется — только минимум и рост по контенту.
6. **Бордеры и дивайдеры**: толщина линии — только **border/default** (1 px) или **border/emphasis** (2 px). Дивайдеры между блоками — **divider/default** (1 px). Не использовать произвольные значения (например 1 px вручную).
7. **Иконки**: в компонентах — только размеры **16**, **24** и **32** px. Крупнее — **illustration/\***, не icon.

8. **Спецификация компонента для разработки**: для каждого компонента в `docs/` — **одна** спека с **таблицей токенов** (все визуальные значения из Semantic / привязка к Core). В таблице указывать:
   - **Отступы** (padding, gap): токены space / stack по ролям.
   - **Радиусы**: токен radius (control-sm, surface и т.д.).
   - **Толщина бордера**: border/default или border/emphasis (если есть обводка).
   - **Текст**: стиль по **TYPOGRAPHY.md** и семантический цвет.
   - **Иконки**: размер 16 / 24 / 32 px и семантический цвет.
   - **Высота контролов**: control-height/xs … control-height/lg.
   - **Размеры не-контролов**: size/2xs … size/2xl.

   **Пример строк таблицы (кнопка):**

   | Параметр           | Токен / значение   |
   |--------------------|--------------------|
   | Padding horizontal | spacing/4          |
   | Padding vertical   | spacing/3          |
   | Gap (icon–text)    | spacing/2          |
   | Radius             | control-md         |
   | Border             | border/default     |
   | Высота             | control-height/md  |
   | Текст              | Body 2, semantic.text.primary |
   | Иконка             | 24 px, semantic.text.primary |

   **Пример строк таблицы (бейдж):**

   | Параметр           | Токен / значение   |
   |--------------------|--------------------|
   | Padding            | spacing/2          |
   | Radius             | pill               |
   | Высота             | size/md            |
   | Текст              | caption-md, semantic.decor[variant].text |

9. **Один компонент — одна спека.** Один `.md` на компонент, без дублирующих файлов.

10. **Порядок для новых компонентов:** спека в `docs/` → реализация в коде. Глобальные правила — здесь и в **TYPOGRAPHY** / **COLOR-PALETTE**; детали контрола — в спеке компонента (пример: **docs/button-spec.md**).

11. **Когда допустим хардкод.** По умолчанию все визуальные значения привязываются к токенам. Хардкод (значение без привязки к токену) допустим **только в редких случаях**, когда токены не решают конкретную задачу:
    - **Оптическая компенсация** — мелкое смещение для выравнивания элементов разной массы (пример: `paddingTop=1` в Compact Alert, `PIN_OFFSET=2dp` в Avatar, ripple-overlay opacity).
    - **Вынужденные размеры не из сетки** — стандартное соотношение требует значения, которого нет на шкале (пример: Inner Dot в Radio = 10×10, ровно 50% от ring 20px; на шкале нет 10, но 8 или 12 визуально хуже).
    - **Декоративные SVG-детали внутри иконок** — внутренние элементы icon library (тонкие линии, акценты в детализированных иконках), которые swap'аются вместе с иконкой и не должны жить отдельной семантикой.

    **Каждый хардкод:**
    - **Документируется** в спеке компонента как design constant с обоснованием — **зачем** именно это значение, **почему** не подошёл ни один токен.
    - **Не повторяется бездумно** — если возникает третий-четвёртый случай одной и той же роли, заводим токен и мигрируем.
    - **Не используется** для лени или импорта чужого UI без локализации (пример: значения из Material 3 — нужно мапить на нашу шкалу, не оставлять `#49454f`).

## Синхронизация с кодом

### Naming в Kotlin / Swift

В Kotlin и Swift идентификатор не может начинаться с цифры — `Size.2xs` не скомпилируется. На этих платформах применяется renaming:

| Figma / CSS | Kotlin / Swift |
|---|---|
| `2xs` | `xxs` |
| `2xl` | `xxl` |

Семантика и численные значения идентичны — это чистое renaming для платформ. На вебе (CSS-переменные) `2xs` / `2xl` доступны как есть.

### Маппинг токенов

| Токен Figma | Код | Значение |
|-------------|-----|----------|
| spacing/N | `spacing[N]` | N × 4 px |
| control-height/* | `controlHeight.xs/sm/md/lg` | 32/40/48/56 px |
| size/* | `size.xxs/xs/sm/md/lg/xl/xxl` (Kotlin/Swift) или `size['2xs']…['2xl']` (TS) | 16/20/24/32/40/48/56 px |
| illustration/* | `illustration.sm/md/lg` | 64/80/120 px |

- `height` — deprecated alias для `controlHeight` (обратная совместимость).
- Семантические имена в коде соответствуют токенам Figma (с учётом renaming `2xs/2xl → xxs/xxl` для строго-типизированных платформ).
